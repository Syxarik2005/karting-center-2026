# Общий план реализации — картинг-центр «Апекс»

> Ранняя версия этого файла была практически дословной копией плана
> референсного проекта «Волна» (SUP-клуб), вплоть до «лимитов досок» вместо
> лимитов картов. Переписан под фактический бриф Дениса и
> `01-analysis/api/openapi.yaml`.

## 1. Бэкенд (Go + PostgreSQL)
- **Фаза 1. Инфраструктура:** Go-модуль (`apex-backend`), конфигурация,
  роутер (chi), генерация SQL-запросов (sqlc) — см. `db_init.sql`,
  `internal/repository/`.
- **Фаза 2. Авторизация:** JWT + мок-OTP по телефону (`pkg/auth`,
  `/auth/send-code`, `/auth/login`). Это дополнительное упражнение по
  «смежным ролям» — контракт клиентского приложения
  (`01-analysis/api/openapi.yaml`) не описывает авторизацию и предполагает
  уже аутентифицированного клиента (R-004/R-028 в customer-brief.md); см.
  пояснение в корневом `README.md`.
- **Фаза 3. Расписание (Read-Only):** `GET /slots`, `GET /slots/{id}` —
  список слотов с фильтром по датам, джоин с маршалами (не «инструкторами
  на маршрутах», как было в черновике).
- **Фаза 4. Бронирование:** Транзакционное создание брони (с проверкой
  лимита картов на слот — BR-01) через Postgres-триггер
  `claim_kart_on_booking`; отмена с порогом по времени (BR-02, см.
  `internal/api/handler.go`, `cancelCutoff`).
- **Фаза 5. Оценка маршала:** `POST /client/bookings/{id}/rate` — один
  отзыв на бронирование, средний рейтинг маршала пересчитывается триггером.

## 2. Клиентское мобильное приложение (Flutter)
- **Фаза 1. Инфраструктура:** Clean Architecture (data/di/presentation),
  роутинг, дизайн-токены (`AppTheme`).
- **Фаза 2. Авторизация:** экран входа по телефону/OTP, сохранение токена
  (`login_screen.dart`, `auth_interceptor.dart`).
- **Фаза 3. Расписание:** список слотов картинга (`SlotModel` — трасса
  SHORT/LONG, маршал, доступные карты), фильтры по периоду/трассе.
  *(Ещё не реализовано в этой поставке — только модели и datasource;
  экраны расписания/бронирования как в `02-development/client` (React)
  здесь предстоит добавить отдельно.)*
- **Фаза 4. Бронирование:** выбор экипировки (OWN/RENTAL) вместо мест на
  борде, история броней, отмена.
- **Фаза 5. Профиль:** `GET /client/profile`.

## Статус на момент сдачи
- Бэкенд: домен переписан под Апекс полностью (схема, sqlc-слой,
  хендлеры, маршруты под `/client/*`), эндпоинты покрывают весь контракт
  `openapi.yaml` + доп. авторизацию.
- Flutter: слой данных (модели, datasource, репозитории) переписан под
  Апекс; экран логина реализован; экраны расписания/бронирования —
  предстоит доделать (см. `02-development/client` — рабочий React-аналог
  этих экранов уже полностью реализован и покрыт тестами, можно
  использовать как референс для Flutter-версии).
