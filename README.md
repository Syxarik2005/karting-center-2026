# 🏎️ Картинг-центр «Апекс» (Системная аналитика)

Добро пожаловать в репозиторий проекта клиентского мобильного приложения для уличного картинг-центра «Апекс»! 

Этот проект выполнен в рамках **Летней Школы Surf**. Здесь собраны артефакты выявления требований и проектирования системы на основе первоначального брифа от владельца бизнеса.

## 📑 Структура проекта

Проект состоит из трёх частей, по дням работы летней школы:

- **`01-analysis/`** — аналитика и проектирование (бриф, требования, дизайн, API-контракт).
- **`02-development/`** — клиентское веб-приложение (React, мобильный форм-фактор), 4 фичи поверх контракта из `01-analysis/api/openapi.yaml`. См. `02-development/README.md`.
- **`03-testing/`** — тест-план, 18 тест-кейсов, 2 найденных и исправленных бага. См. `03-testing/README.md`.

Дополнительно, в порядке практики «смежных ролей» (бэкенд/мобильная разработка), в репозитории также лежат:
- **`cmd/`, `internal/`, `pkg/`, `db_init.sql`** — экспериментальный Go-бэкенд под тот же домен.
- **`mobile_app/`** — экспериментальное Flutter-приложение (слой данных переписан под Апекс, экраны расписания/бронирования ещё не дореализованы — референс для них уже есть в `02-development/client`).

**Важно:** `02-development/client` (React) — самостоятельная, полностью рабочая
реализация клиента поверх `01-analysis/api/openapi.yaml`, где бэкенд
трактуется как чёрный ящик (см. ниже). Go-бэкенд и Flutter-приложение — это
отдельное, дополнительное упражнение «а что если бы мы сами писали и бэкенд»,
не обязательное по контракту `openapi.yaml` и не заменяющее React-реализацию.

Содержимое `01-analysis/`:

- 📁 **`0-customer-brief/`** — Исходный "живой" бриф от Дениса, владельца картинг-центра.
- 📁 **`1-elicitation/`** — Выявление требований. Содержит описание предметной области (Domain description) и стратегические вопросы к заказчику.
- 📁 **`2-requirements/`** — User stories, сценарии и acceptance-критерии MVP.
- 📁 **`3-design-brief/`** — Бриф на дизайн клиентского приложения.
- 📁 **`4-design/`** 
  - `screens/screen-registry.md` — Полный реестр экранов мобильного приложения и их стейтов.
  - `er-model-and-sequences.md` — ER-модель клиентской части и Sequence diagram для процесса бронирования заезда (учитывающая все краевые случаи: отмена, нехватка мест).
- 📁 **`5-mobile-app-spec/`** — (Зарезервировано) Детальные спецификации логики приложения.
- 📁 **`api/`** — Контракт взаимодействия с бэкендом (OpenAPI Specification в формате YAML).
- 📁 **`prompts/`** — Промпты, использованные при генерации аналитики.

## 🎯 Ключевые особенности домена

1. **Бэкенд как Black Box (для `02-development/client`):** React-клиент не управляет админкой, расписанием или маршалами — только потребляет готовые данные из `openapi.yaml` и отдаёт команды на бронирование. Экспериментальный Go-бэкенд в этом репозитории существует параллельно, как отдельное упражнение, и не отменяет этого допущения для основной поставки.
2. **Атомарность:** Обработка отказов при одновременной записи (ловушка 409 Conflict).
3. **Отмены и погода:** Поскольку картинг уличный, учтён сценарий внезапной отмены заездов администрацией из-за непогоды (с пуш-уведомлением клиентам).
