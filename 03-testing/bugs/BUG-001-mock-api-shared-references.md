# BUG-001 — Мок-API возвращал общие ссылки на объекты вместо снапшотов

## Симптом / цель
Написал тест «бронирование уменьшает `available_karts` на 1» и он падал с
`expected 4 to be 3`, хотя логика уменьшения в `createBooking` выглядела
правильной на глаз (`slot.available_karts -= 1`).

## Требования (что должно быть)
Реальный HTTP-ответ — это всегда независимая копия данных на момент запроса
(JSON сериализуется и десериализуется заново). Если клиент сохранил слот
«до» операции, он не должен внезапно увидеть, что это состояние задним числом
изменилось — так работает любой настоящий REST API, и мок обязан вести себя
так же (иначе тестирование клиентской логики становится нечестным: баги,
которые проявились бы на реальном бэкенде через рассинхрон кэша, здесь просто
не воспроизводятся).

## Причина
`getSlots`/`getSlot`/`getMyBookings`/`getBooking` возвращали **сами объекты**
из массива `slotsDb`/`bookingsDb`, а не их копии. `createBooking` и
`cancelBooking` мутируют эти объекты напрямую (`slot.available_karts -= 1`).
В результате любой код, который взял слот через `getSlots()` **до** брони и
сравнил его с состоянием **после** — на самом деле сравнивал один и тот же
объект сам с собой, потому что ссылка не менялась.

Из-за этого в `SlotDetailsPage` теоретически мог быть трудноуловимый баг:
если бы где-то в коде кэшировался слот и сравнивался с обновлённым (например,
для анимации «−1 место»), сравнение всегда показывало бы «изменений нет».

## Исправление
Добавлены `cloneSlot`/`cloneBooking` в `src/api/mockApi.ts` — каждая
read-функция (`getSlots`, `getSlot`, `getMyBookings`, `getBooking`) и
успешный `createBooking`/`cancelBooking` теперь возвращают **копию**, а не
ссылку на внутреннее состояние. Внутренняя мутация (`slotsDb`/`bookingsDb`)
осталась прежней — меняется только то, что отдаётся наружу.

## Промпты (переданные ИИ-ассистенту)
```
1. "Тест 'succeeds and decrements available_karts' падает: expected 4 to be
    3. before и after — это, кажется, один и тот же объект. Проверь, не
    возвращает ли мок живые ссылки на slotsDb вместо копий."

2. "Заведи cloneSlot/cloneBooking и прогони через них каждый return в
    mockApi.ts, чтобы ни один вызов не отдавал наружу объект, который потом
    может быть тихо изменён другой функцией."
```

## Ручная проверка
1. `npm test` в `02-development/client` — кейс `succeeds and decrements
   available_karts` зелёный.
2. В браузере: открыть `/`, запомнить число мест у любого слота, открыть его
   и записаться, вернуться на `/` — число должно быть на 1 меньше **и**
   старая переменная (если бы её сохранили в devtools console) не должна
   была измениться сама по себе.
