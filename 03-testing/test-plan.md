# Тест-план — клиентское приложение «Апекс»

> Часть 3 задания летней школы: «Сгенерировать тест-кейсы, найти и исправить
> от 1 до 3 багов». Тесты написаны на Vitest + Testing Library и реально
> запускаются (не просто список кейсов на бумаге).

## Что покрыто и почему именно так

Приоритет отдан **бизнес-правилам** (`domain-description.md`, BR-01..BR-04) и
**обработке ошибок**, а не проверке вёрстки — потому что именно тут дороже
всего словить баг в проде: конфликт бронирования, неправильная отмена,
показ чужого состояния.

| Файл | Что проверяет | Уровень |
| --- | --- | --- |
| `src/api/mockApi.test.ts` | Контракт мок-API: горизонт 7 дней (BR-03), фильтр по трассе, атомарность бронирования (BR-01) — 409/410, порог отмены и возврат карта, отмена центром (BR-02), оценка маршала | Юнит (13 тест-кейсов) |
| `src/pages/SlotDetailsPage.test.tsx` | Кнопка недоступна при 0 мест; честный конфликт 409 показывается текстом, а не белым экраном | Компонентный (2) |
| `src/pages/BookingDetailsPage.test.tsx` | Успешная отмена вне порога; понятная ошибка и **сохранение статуса "Активна"** при попытке отмены внутри порога | Компонентный (2) |
| `src/pages/MyBookingsPage.test.tsx` | Новая активная бронь попадает во вкладку "Предстоящие", а не "Прошедшие" | Компонентный (1) |

Итого: **18 тест-кейсов**, все проходят.

## Как запустить

```bash
cd 02-development/client
npm install    # если ещё не делали
npm test       # vitest run — разово, без watch-режима
```

## Тест-кейсы, которые специально написаны, чтобы ЛОМАТЬСЯ, если баг вернётся

- `succeeds and decrements available_karts` — именно этот кейс поймал BUG-001
  (см. `bugs/BUG-001-mock-api-shared-references.md`): без фикса `before` и
  `after` оказывались одним и тем же объектом.
- `rejects cancellation inside the cutoff window with a 400` — использует
  отдельный `slot-9-near-term` (старт через 10 минут) специально, чтобы порог
  отмены был проверяем без обращения к приватному состоянию модуля.

## Что сознательно НЕ покрыто в этой поставке

- Настоящая конкурентная гонка (два реальных пользователя жмут "Записаться"
  одновременно) — мок синхронный, реальную гонку можно проверить только на
  бэкенде с БД; здесь симулируется через ручное создание брони между загрузкой
  страницы и сабмитом (см. `SlotDetailsPage.test.tsx`, второй кейс).
- Экран профиля (SCR-006) — read-only без бизнес-логики, тестировать нечего
  сверх самого факта рендера.
- Визуальная регрессия (скриншот-тесты) — вне скоупа учебного проекта.
