# FEAT-003 — Мои заезды и отмена брони (SCR-003, SCR-004)

## Цель
Клиент видит свои брони (предстоящие/прошедшие), может отменить активную
бронь с подтверждением, и явно видит причину, если бронь отменил центр
(погода/оргпричины) — без возможности перезаписаться на тот же слот.

## Требования
- `customer-brief.md` — «иногда люди отменяются… если за 10 минут до старта
  — это уже проблема, надо как-то это ограничить».
- `domain-description.md` — открытый вопрос №1 (нет точного порога отмены от
  заказчика) + BR-02 (отмена центром: статус, причина, push, запрет повторной записи).
- `screen-registry.md` — SCR-003 (вкладки Предстоящие/Прошедшие), SCR-004
  (кнопка «Отменить запись» только при статусе Активно).
- `openapi.yaml` — `GET /client/bookings`, `POST /client/bookings/{id}/cancel`,
  ответ `400` при попытке отменить слишком поздно.

## Реализация
- `src/pages/MyBookingsPage.tsx` — две вкладки (фильтр по `status === 'ACTIVE'`
  vs остальное), пустые состояния под каждую вкладку отдельно.
- `src/pages/BookingDetailsPage.tsx` — детали брони + кнопка отмены с явным
  шагом подтверждения (не сразу DELETE по одному тапу), плашка с причиной для
  `CANCELLED_BY_CENTER`.
- `src/api/mockApi.ts` → `cancelBooking`:
  - считает `minutesToStart` от текущего момента;
  - если меньше `CANCEL_CUTOFF_MINUTES` (сейчас **60**, см. «Допущение» ниже) —
    кидает `400 CANCEL_TOO_LATE`;
  - при успехе возвращает карт в `available_karts` слота (симметрично FEAT-002).

## ⚠️ Допущение, требующее подтверждения у Дениса
В брифе нет точной цифры порога отмены — только упоминание «10 минут» как
проблемного случая. Взял **60 минут** как безопасный дефолт для демо (учебный
проект), а не «10 минут», потому что 10 минут не оставляет центру времени
предложить карт следующему в очереди. Зафиксировано как открытый вопрос №1 в
`domain-description.md` — это решение аналитика/заказчика, не моё.

## Промпты (переданные ИИ-ассистенту)
```
1. "SCR-003 — две вкладки (предстоящие/прошедшие), каждая со своим пустым
    состоянием. SCR-004 — кнопка отмены только если status ACTIVE, с шагом
    подтверждения перед реальной отменой."

2. "Реализуй порог отмены как константу CANCEL_CUTOFF_MINUTES, не хардкодь
    число в трёх местах, и явно закомментируй, что это допущение, а не факт
    из брифа — потому что в брифе конкретной цифры нет."

3. "После отмены брони — верни карт обратно в available_karts мок-слота,
    чтобы список расписания не расходился со списком броней."
```

## Ручная проверка
1. `/bookings` → вкладка «Предстоящие» — там нет `booking-1`
   (CANCELLED_BY_CENTER) и `booking-2` (COMPLETED), они только на вкладке
   «Прошедшие/Отменённые».
2. Открыть `booking-1` → должна быть видна плашка «Отменена центром» с
   причиной «Отмена из-за гололёда на трассе», без кнопки отмены.
3. Забронировать слот через FEAT-002 (старт через несколько часов, > 60 минут)
   → на его странице брони кнопка «Отменить запись» активна → подтвердить →
   статус меняется на «Отменена вами», и в расписании у слота снова +1 место.
4. Проверить порог: если бы старт был раньше, чем через 60 минут — отмена
   должна была бы вернуть 400 и текст с точным порогом (легко проверить,
   временно понизив `hoursFromNow` у тестового слота).
