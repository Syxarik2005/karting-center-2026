# Архитектурный план — клиентское приложение «Апекс»

> Часть 2 задания летней школы: «Получить от ИИ архитектурный план и схему данных».
> Входные данные: `01-analysis/0-customer-brief`, `01-analysis/1-elicitation/domain-description.md`,
> `01-analysis/4-design/screen-registry.md`, `01-analysis/api/openapi.yaml`.

## 1. Выбор формы реализации

Заказчик просил мобильное приложение, но всё, что он реально описывает — это простой
CRUD-флоу поверх готового API (посмотреть слоты → записаться → посмотреть свои брони →
отменить/оценить). Для демонстрации в рамках летней школы выбран **React web-app,
мобильный форм-фактор** (max-width контейнер, нижняя таб-навигация, тач-friendly
контролы) — это тот же UX, что в нативном приложении, но без накладных расходов на
нативную сборку. Если проект пойдёт дальше учебного контекста, это же дерево компонентов
переносится в React Native почти 1:1 (бизнес-логика и API-слой не завязаны на DOM).

## 2. Стек

| Слой | Выбор | Почему |
| --- | --- | --- |
| Сборка | Vite + React 19 + TypeScript | быстрый старт, HMR, типы защищают от рассинхрона с OpenAPI-контрактом |
| Роутинг | react-router-dom v7 | нужен полноценный стек экранов (список → детали → бронирование → мои брони) |
| Серверное состояние | @tanstack/react-query | кэширование, `invalidateQueries` после мутаций (создание/отмена брони, оценка) — ровно то, что нужно для этого приложения, без Redux |
| Стили | Tailwind CSS v4 | скорость вёрстки мобильных карточек/состояний без отдельного дизайн-файла |
| API-слой | `src/api/*` — типы + мок-клиент | бэкенд вне скоупа (R-004); мок воспроизводит контракт 1:1, включая коды ошибок 409/410/400, чтобы фронтенд-логика была честной, а не «на глаз» |

## 3. Структура проекта

```
02-development/client/
├── src/
│   ├── api/
│   │   ├── types.ts       # 1:1 с components.schemas из openapi.yaml
│   │   ├── mockData.ts    # сид-данные (слоты, маршалы, брони, профиль)
│   │   └── mockApi.ts     # имитация эндпоинтов + бизнес-правил (BR-01..BR-04)
│   ├── components/        # переиспользуемые UI-блоки (карточки, состояния, нав)
│   ├── pages/              # по одной странице на экран из screen-registry.md
│   ├── utils.ts            # форматирование дат/статусов
│   ├── App.tsx              # роутинг + общий layout
│   └── main.tsx              # провайдеры (QueryClient, BrowserRouter)
```

Разделение `api/` от `pages/` — сознательное: когда бэкенд появится, меняется только
`mockApi.ts` → `realApi.ts` (те же сигнатуры функций), страницы и компоненты не трогаем.

## 5. Обработка бизнес-правил на клиенте

- **BR-01 (атомарность бронирования):** `createBooking` в моке кидает `409` если
  `available_karts <= 0` в момент отправки (не в момент открытия экрана) — это
  специально воспроизводит гонку, которую должен обрабатывать реальный клиент.
  UI обязан инвалидировать кэш слота/списка после конфликта, а не просто показать тост.
- **BR-02 (отмена центром):** статус `CANCELLED_BY_CENTER` не даёт повторно
  забронировать слот и всегда показывает причину — проверено в `BookingDetailsPage`.
- **BR-03 (горизонт 7 дней):** `ScheduleListPage` запрашивает `date_to = now + 7d` по
  умолчанию, фильтр по периоду — явное расширение.
- **BR-04 (без онлайн-оплаты):** экран бронирования показывает тариф проката как
  информацию, кнопки оплаты нет нигде.
